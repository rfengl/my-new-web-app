1 Introduction

1.1 Purpose

Volvo provided CarFix with 15 OpenAPI specifications and asked us to integrate our call center to deliver Roadside Assistance to Volvo drivers. This document is CarFix's MVP study — it scopes the minimum integration that allows the CarFix call center to:

  1. Receive a vehicle-initiated bCall / iCall.
  2. Identify the caller and the vehicle.
  3. Track the active xCall service through to closure.
  4. Observe live vehicle state to inform dispatch.
  5. Trigger the vehicle alarm if needed (panic-light scenario).

Anything beyond that — door unlock, ownership verification at API level, location nudges, EV charging history, start-inhibit, engine control, POI, fleet provisioning — is excluded from MVP and tracked in the parent study.

1.2 Scope

In scope for MVP

  • Receiving vehicle-initiated emergency / breakdown events (bCall, iCall) at the CarFix call center.
  • Identifying the caller's vehicle and customer from VIN, plate, or MSISDN.
  • Managing the lifecycle of an active xCall service from the CarFix agent desktop.
  • Triggering or querying the vehicle alarm (RAM) — the only remote command in MVP scope.
  • Live vehicle status (fuel / EV charge level, doors, ignition, mileage) and last-known position via DAM.
  • Real-time push and / or long-poll consumption of WirelessCar notifications.
  • One-time tenant notification configuration.
  • GDPR / data-subject handling for any personal data persisted on CarFix systems.

Out of MVP scope (deferred — see parent study)

  • API-level ownership verification (/vehicles/{id}/user-relations). For MVP, the agent verifies the caller's identity verbally during the call; the API-level gate is added in a later phase.
  • Remote door unlock (RDL), honk-and-flash (RHF), POI nav, RCS / RDC, start-inhibit (RSI), engine control (REC) — deferred pending elevated scope grants and liability sign-off.
  • EV charging history (ev-charging-log-service.yml) — deferred until EV market scope is confirmed.
  • Fleet provisioning via DAM Admin (POST /fleets etc.) — for MVP we assume Volvo / WirelessCar provisions CarFix's fleet for us during onboarding; we do not exercise the admin API.
  • Vehicle telematics ingestion (Volvo / WirelessCar own this end-to-end).
  • Volvo's own customer-facing apps (Volvo On-Call, Volvo Cars app).
  • CarFix's own tow-operator dispatch, payment, and CRM systems.
  • The fleet-authentication-api.yml token issuer — COMPANY_INTERNAL, not the issuer for external partners (Q-01).

2 System Overview

2.1 Service description

CarFix delivers roadside assistance (towing, jump-start, fuel delivery, lock-out recovery, accident response) to Volvo drivers covered by Volvo's service contract. WirelessCar is the telematics intermediary: it relays vehicle-emitted xCall events to CarFix, exposes vehicle and customer master data, and proxies the alarm command back to the vehicle. The CarFix call center never speaks directly to the vehicle — every interaction flows through the WirelessCar API surface.

Lock-out recovery, locate-and-honk, POI nudges, and engine-control remediations are part of CarFix's longer-term service offering but are not part of this MVP — for MVP, those scenarios are handled by on-site dispatch (tow / locksmith) rather than remote command.

4 Functional Requirements (MVP)

FR-IDs are preserved from the parent study for traceability — gaps (FR-006, FR-009, FR-010, FR-012 – FR-016) are intentional and indicate the requirement is deferred post-MVP. Each requirement is identified by an FR-NNN tag with a MoSCoW priority. The API source row points at the specific YAML and endpoint(s) that satisfy the requirement.

FR-001 M2M Authentication Must
Description
Obtain a short-lived JWT bearer token used to authenticate every other call into the WirelessCar surface.
Trigger
Token cache miss, expiry, or scheduled secret rotation.
Inputs
client_id, client_secret, grant_type=client_credentials, target audience, tenant, plus per-action scopes such as Read_RAM, Write_RAM.
Processing
Cache tokens until exp − 60 s; refresh proactively; verify any inbound JWT using the JWKS at GET /oauth/.well-known/jwks.json.
Outputs
A signed JWT presented as Authorization: Bearer <token> on every downstream call.
Validation
Token must validate against the regional JWKS; audience and scope claims must match the target endpoint family.
API source
access-management-m2m-api.yml — POST /v2/oauth/token access-management-m2m-api.yml — GET /oauth/.well-known/jwks.json
Note
fleet-authentication-api.yml is not the right issuer for CarFix — it is x-audience: COMPANY_INTERNAL. Confirm Q-01.
FR-002 Subscribe to vehicle downlink events Must
Description
Register a subscription so vehicle-initiated events (bCall, iCall, status, alerts) are routed to CarFix.
Trigger
One-time during onboarding; subsequent updates only when filter criteria change.
Outputs
A subscriptionId to be used by FR-003 / FR-008.
API source
call-center-services-downlink-subscriptions-api.yml — POST /subscriptions, GET /subscriptions, PUT /subscriptions/{subscriptionId}, DELETE /subscriptions/{subscriptionId}, POST /subscriptions/{subscriptionId}/confirmation
FR-003 Receive inbound bCall / iCall Must
Description
CarFix consumes vehicle-initiated emergency and information calls and acknowledges receipt so they are not re-delivered.
Trigger
A vehicle emits bCall (driver SOS, fault) or iCall (information request); WirelessCar enqueues to the CarFix downlink topic.
Processing
  • Long-poll the notification endpoint; the call returns within ~20 s, with 0 – 10 messages per response.
  • Each message carries serviceId, call type (bCall / iCall), vehicle identifiers, location, occupants, vehicle state snapshot, and a receiptHandle.
  • Acknowledge each handled message within the visibility window.
Outputs
Persisted CarFix case keyed on serviceId.
API source
call-center-services-xcall-downlink-notification-api.yml — GET /notifications/{subscriptionId}, PUT /notifications/{subscriptionId}/acknowledgement
FR-004 Manage active call lifecycle Must
Description
CarFix agent updates, terminates, or clears an active xCall service; may also create a service from the partner side.
API source
call-center-services-xcall-consumer.yml — POST /services/started, GET /services/{serviceId}, GET /services/{serviceId}/events, PUT /services/{serviceId}/updated, PUT /services/{serviceId}/terminated, PUT /services/{serviceId}/cleared
Validation
State transitions follow the CCS service-state machine. Terminal states are immutable.
FR-005 Identify vehicle and customer Must
Description
Look up a vehicle by VIN, registration plate, or embedded SIM number; resolve the registered driver(s) for display on the agent desktop.
Trigger
Inbound call answered (from FR-003) or customer phones in manually.
Inputs
One of vehicleIdType ∈ {VEHICLE_IDENTIFIER, VIN, MSISDN} + identifier value, OR a free-text plate via search.
API source
call-center-services-master-data-adapter.yml — GET /vehicles/{vehicleId}, POST /search/vehicles, GET /vehicles/{vehicleId}/services, GET /users/{userId}, POST /search/users
MVP note
The agent uses this data to display the registered driver alongside the caller-stated identity for verbal verification. The API-level ownership cross-check (/vehicles/{id}/user-relations) is post-MVP — see parent study FR-006.
FR-007 Real-time vehicle status Should
Description
Retrieve current door-lock state, fuel / EV charge, mileage, engine state, and last known position — used by the agent to inform dispatch (e.g. fuel-low → fuel delivery, doors-locked → locksmith).
Region
DAM ships EU + CN servers only; NA usage requires Volvo confirmation (Q-10).
API source
data-access-management-api.yml — GET /fleets/{fleetIdentifier}/vehicles/{vehicleIdentifier}/status, .../position, bulk variants, /info, /vehicles, /trips
MVP note
CarFix reads vehicle status; no remediation command is issued in MVP except RAM alarm (FR-011). All other remote actions are dispatched on-site.
FR-008 Real-time push of state changes Should
Channel choice
  • WebSocket push — wss://api.notifications.call-center-services.* via call-center-services-notifications-manager.yml ($connect, $disconnect, publish, acknowledge).
  • HTTP long-poll — call-center-services-consumer-notifications.yml (GET /notifications/{subscriptionId}, PUT .../acknowledgement) paired with call-center-services-consumer-subscriptions-api.yml.
MVP rec.
Long-poll for first launch; WebSocket evaluated post-MVP if latency demands it. Q-05 to Volvo for confirmation.
FR-011 Trigger or query alarm (RAM) Should
Description
Trigger or query the vehicle alarm — used in SOS / panic-light scenarios where the driver needs to draw attention to a parked vehicle (e.g. theft suspicion or lost-vehicle locate at low risk). The only Remote Service command family in MVP scope.
Inputs
vehicleId (resolved via FR-005), command body per spec.
Processing
POST the command; poll GET .../RAM/commands/{commandId} until terminal state (Success / Failure / Timeout); never auto-retry on Timeout.
API source
call-center-services-remote-service.yml — POST /vehicles/{vehicleId}/services/RAM/commands call-center-services-remote-service.yml — GET /vehicles/{vehicleId}/services/RAM/commands/{commandId}
Validation
Requires Read_RAM / Write_RAM scope provisioned by WirelessCar. MVP guardrail: the command is gated at the CarFix UI to a verified active xCall serviceId (i.e. the driver is on the line) — API-level ownership verification is post-MVP.
FR-017 Tenant notification configuration (one-time) Should
Description
Configure how CarFix receives consumer notifications for the tenant.
API source
call-center-services-tenant-configuration-admin.yml — GET / POST / PUT / DELETE /configurations/consumer/notifications
FR-018 GDPR data-subject requests Could
Description
Service Volvo data controllers' requests for personal data export or purge that touch the CarFix-side store.
API source
call-center-services-privacy-api.yml — GET /vehicles/{vehicleId} (initiate fetch — returns receipt) call-center-services-privacy-api.yml — POST /vehicles/{vehicleId} (purge), GET /receipts/{receiptId} (status / download)
FR-019 Client secret rotation Must
Description
Rotate CarFix's M2M secret at least every 90 days, or on suspected compromise (Q-04 to Volvo).
API source
access-management-m2m-api.yml — POST /secret-rotation/client-secret-create (expires_in 30 – 365 days) access-management-m2m-api.yml — DELETE /secret-rotation/client-secret-delete
Validation
Rotation MUST overlap (create new → cut over → delete old).
Excluded from MVP (see parent study): FR-006 ownership gate, FR-009 RDL door unlock, FR-010 RHF honk-and-flash, FR-012 RSI / REC start-inhibit and engine control, FR-013 POI navigation, FR-014 RCS connectivity / RDC door control, FR-015 EV charging history, FR-016 fleet provisioning.

6.4 Remote-command status state machine (RAM)

   POST .../RAM/commands
        │
        ▼
     Pending ──── Timeout (no vehicle response in window)
        │
        ▼
    Success | Failure
Pending is always the initial response. Poll GET …/RAM/commands/{commandId} until a terminal state. On Timeout, the actual vehicle-side outcome is indeterminate; do not retry blindly.

8 Validation & Business Rules (MVP)

RuleBehaviour
Bearer token freshnessRefresh before exp − 60 s; refuse to issue any call if no valid token can be obtained.
Audience matchThe audience claim of the issued token MUST match the API family being called.
Per-action scopeThe RAM command requires Read_RAM / Write_RAM scope; missing scope ⇒ 403.
At-least-once dedupAll inbound notifications deduplicated on serviceId + event sequence number.
Command status pollingInitial response is always Pending; only Success / Failure / Timeout are terminal. No blind retry on Timeout.
Identity gate (MVP)API-level ownership verification is post-MVP. The agent verbally confirms caller identity, and a RAM command requires an active serviceId from a vehicle-initiated bCall (i.e. the driver is on the line).
Region pinningA vehicle is served only from the WirelessCar region matching its registration; cross-region calls are not supported.
Tracing propagationtraceparent is a hard requirement on all outbound calls.

9 Error Handling

ClassBehaviour
401 / 403 — auth / scopeRefresh token; if still 401, alert ops. 403 = scope misconfig — not retryable.
404 on vehicleId / serviceIdSurface to agent as "vehicle / case not found"; do not retry.
409 / state conflict on xCall lifecycleRe-fetch GET /services/{serviceId} and reconcile UI state.
RAM command TimeoutOutcome unknown — do not auto-retry. Escalate to agent for manual confirmation.
Notification ack deadline missedMessage will be redelivered; dedup logic in NFR-006 must prevent double-processing.
WebSocket drop (FR-008)Reconnect with exponential backoff (jitter); fall back to long-poll after N consecutive failures.
JWKS fetch failureServe from cache up to TTL; if expired, fail-closed on inbound JWT verification.

10 Acceptance Criteria (MVP)

AC-IDs preserved from the parent study for traceability — gaps (AC-005, AC-006, AC-007) indicate criteria deferred post-MVP because they target the dropped FR-006 / FR-009 surface.

AC-001 M2M token issuance (FR-001)
Given a registered CarFix tenant with valid client_id / client_secret When the CarFix backend POSTs to /v2/oauth/token with audience WirelessCar/CCS Downlink Notification Then a JWT is returned, validates against the regional JWKS, and carries the requested audience and scope claims.
AC-002 Inbound bCall reception (FR-003)
Given an active downlink subscription for the CarFix tenant When the test vehicle emits a bCall Then the long-poll receives the message within ≤ 20 s, with serviceId, vehicle identifiers, position, and occupant count.
AC-003 At-least-once dedup (NFR-006)
Given a notification fetched but not acknowledged When the visibility window expires and WirelessCar redelivers it Then CarFix detects the duplicate via serviceId and creates no second case.
AC-004 Vehicle lookup by VIN (FR-005)
Given the agent has a 17-character VIN When the agent submits the lookup Then GET /vehicles/{id}?vehicleIdType=VIN returns the canonical vehicle record within ≤ 1 s p95.
AC-008 xCall lifecycle update (FR-004)
Given an active serviceId When the agent posts an update via PUT /services/{serviceId}/updated Then a subsequent GET /services/{serviceId} reflects the new state.
AC-009 Trace propagation (NFR-003)
Given an inbound notification carrying a traceparent When CarFix makes any downstream WirelessCar call for the same case Then the same trace ID appears in CarFix's call logs and on WirelessCar's side.
AC-010 Region isolation (NFR-002)
Given a vehicle registered in EU When CarFix accidentally calls the CN base URL with the EU token Then WirelessCar rejects the call (401 / 403); CarFix's routing layer must prevent this in production.
AC-011 Secret rotation overlap (FR-019)
Given a scheduled rotation When CarFix creates a new secret then continues to issue tokens for ≥ 60 s before deleting the old secret Then no in-flight integration call fails during the rotation window.
AC-012 RAM alarm happy path (FR-011) — new for MVP
Given an active serviceId from a vehicle-initiated bCall When the agent triggers the RAM alarm Then the command transitions Pending → Success within the WirelessCar SLA, and the agent is notified.
AC-013 RAM alarm timeout (FR-011 / NFR-006) — new for MVP
Given a vehicle with poor connectivity When the RAM command transitions to Timeout Then CarFix does not auto-retry; agent is prompted to re-attempt manually; case event log records the timeout.

12 Glossary

TermContext
Audience claimThe audience value passed at token-issuance time and present in the issued JWT; WirelessCar uses it to gate which API family the token may call.
TenantCarFix's logical identity within WirelessCar. Every call carries an implicit or explicit tenant scoping.
Long-pollThe HTTP pattern used by xCall Downlink Notification and Consumer Notifications: a GET returns up to 10 messages or waits ~20 s before returning empty.
AcknowledgementThe PUT …/acknowledgement call that prevents WirelessCar from redelivering an already-handled message.
Service / serviceIdA single CarFix case, originated by a vehicle bCall / iCall or by an agent-initiated POST /services/started.
Command / commandIdA RAM-family request issued by CarFix against a vehicle; status is polled until terminal.
Elevated scopePermissions (e.g. Write_RDL, Write_RSI, Write_REC) granted by WirelessCar only after partner liability sign-off — not in MVP. MVP uses only Read_RAM / Write_RAM.
MoSCoWPrioritisation scheme: Must, Should, Could (this study does not use the Won't tier).
At-least-onceDelivery semantics for WirelessCar notifications: every event is delivered ≥ 1 time; the consumer is responsible for dedup.
